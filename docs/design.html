📱 업무 계획관리 앱 — 최종 설계도

이케다 지에 「매일 아침 1시간이 나를 바꾼다」 방법론 기반 · 최종 갱신 2026-07-20

1. 앱의 정체성 (한 문장)

이케다 지에의 '아침 1시간' 방법론을 기반으로, 매일 우선순위를 정하고 하루 일정을 시간순으로 계획하며, 자유롭게 생각을 쏟아내 새로운 할 일·아이디어를 발굴하고, 매일의 습관을 꾸준히 쌓아가도록 돕는 업무 계획관리 앱.

최종 목표: 외부(시중) 배포 가능한 상용 수준 품질. 우선은 회사 직원 대상. 회사는 태블릿(펜 지원) 보급 환경. B2C 기업이라 일반적으로 회사 프로젝트 업무 중심.

2. 기술 방향

  • 프레임워크: Flutter (Dart) — 안드로이드 우선, 추후 iOS 확장 가능
  • 편집기: VS Code + Flutter 확장 / 빌드·테스트용 Android Studio
  • 개발 PC: Windows 64비트 (개인 PC 또는 회사 팀 노트북에서 작업)
  • 대상 기기: 태블릿 우선. 화면은 폰·태블릿 모두 반응형으로 설계. 펜 입력 고려
  • 데이터 저장: 1단계는 기기 로컬 저장(Hive) → 추후 로그인·클라우드 동기화 확장
  • 주요 패키지: provider(상태관리), hive·hive_flutter(로컬 저장), shared_preferences(다크모드·성장 동물 설정), intl(날짜 포맷), uuid(고유 ID), table_calendar(월간 캘린더). 개발용: build_runner, hive_generator.
  • ※ Power Apps와는 코드 호환이 안 됨(별개 결과물). 향후 Teams 활용이 필요하면 Flutter 웹 버전을 Teams 탭으로 삽입하는 경로를 검토.

3. 핵심 사용자 흐름

아침에 앱을 연다 → 오늘의 일정·할 일을 시간순으로 한눈에 본다 → 오늘 할 일을 적는다 → 중요도·긴급도로 분류(2×2 매트릭스) → 오늘 집중할 3가지 선정 → 하루 동안 완료 체크 → 저녁에 회고한다
아침 1시간 루틴: '아침' 탭에서 타이머를 켜고 집중 시간(기본 60분)을 측정 → 끝나면 무엇을 했는지 메모와 함께 기록 저장 → 실천한 날의 연속일수가 쌓임
생각 노트: 부담 없이 쏟아내기(빈 종이) → 나중에 분류 → '할 일로 전환' 버튼으로 실제 태스크와 연결
습관·챌린지: 반복하고 싶은 습관을 등록 → 매일 '오늘 했음' 체크 → 연속일수(🔥)가 쌓이며 꾸준함을 시각적으로 확인 → 주간 목표·달력 뷰·통계로 심화 관리
전체 일정 보기: 오늘 화면 헤더의 달력 버튼 → 월간 캘린더에서 날짜별 할 일 개수를 점으로 확인 → 날짜를 골라 그날 일정으로 바로 이동
성장·동기부여: 완료한 할 일·습관 체크·아침 세션이 점수로 쌓임 → 통계 대시보드에서 예스코 마스코트가 알→아기→성체→다 큰→마스터로 성장 → 요약·7일 그래프·카테고리별 완료로 성취를 한눈에 확인
프로젝트 관리: 오늘 화면 우측 상단 폴더 버튼 → 프로젝트를 만들고(이름·색상·기간) 할 일에 프로젝트를 연결 → 프로젝트별 진행률(속한 할 일 완료 비율)을 카드로 확인 → 타임라인(간트) 뷰에서 기간 막대·진행률·오늘 위치를 한눈에 확인

성장 마스코트(예스코) 2026-07-17 · 모델 없음(계산·설정값)

항목설명

마스코트 이미지assets/growth/ 의 egg.png(알) + yesco1~4.png(아기~마스터). 회사 마스코트 예스코 사용. growthImagePath가 단계에 맞는 경로 반환.

중요 1: 생각 노트는 지금 비워두더라도 '펜 그림'·'변환된 텍스트' 칸을 데이터 구조에 미리 열어둔다. 나중에 구조를 뜯어고치지 않기 위함.
중요 2: Hive typeId는 서로 겹치면 안 됨 → Task=0, Note=1, NoteStatus=2, Reflection=3, Habit=4, MorningSession=5, Project=6. 새 모델 추가 시 7번부터 사용할 것. (성장 마스코트는 새 모델 없이 기존 데이터 계산으로 구현했으므로 typeId를 쓰지 않음)

5. 화면 구성 (현재까지 구현)

  • 오늘 화면(홈): 오늘의 일정을 시간순 리스트로 표시(시간 있는 일은 시간순, 없는 일은 '시간 미정' 묶음). 상단에 '오늘 집중 3가지' 강조. 날짜를 앞뒤로 넘겨볼 수 있음. 목록 위에 장소 필터(전체/집/외부)와 카테고리 필터(전체/업무/부업/개인/투자, 가로 스크롤) 두 줄, 각 할 일에 장소 아이콘(🏠/🚶)과 카테고리 색상 뱃지 표시. 헤더 우측에 달력 버튼·프로젝트(폴더) 버튼·통계 버튼. 프로젝트 버튼 2026-07-18 추가
  • 프로젝트 목록 화면: 프로젝트를 카드로 표시(색상 점 + 이름 + 완료/전체 할 일 수 + 진행률 바 + % + 설명). 우하단 + 버튼으로 추가(이름·설명·색상 8종·시작일·마감일 입력), 카드 길게 누르면 메뉴(완료 토글 / 수정 / 삭제). 삭제 시 속한 할 일은 지우지 않고 미분류로 남김. 헤더 우측 타임라인 버튼으로 간트 뷰 진입. 2026-07-18 신설
  • 프로젝트 타임라인(간트) 화면: 기간(시작일·마감일)이 설정된 프로젝트를 가로 막대로 표시. 하루 ≈ 20px, 전체 범위는 최소 시작일~최대 마감일 + 앞뒤 3일 여백으로 자동 계산, 가로 스크롤. 막대는 연한 배경(전체 기간) 위 진한 색으로 진행률만큼 채워지고 % 표시, 오늘 날짜에 빨간 세로선, 상단 7일 간격 날짜 눈금. 기간 미설정 프로젝트는 아래쪽에 따로 나열. 2026-07-18 신설

7. 확정된 설계 원칙

  • MVP는 "작동하는 가장 작은 완성품". 다만 난이도가 비슷한 기능은 함께 만들어 불필요하게 미루지 않는다.
  • 코드 바깥이 얽히는 기능(알림·펜 변환·로그인·클라우드)은 반드시 분리해 뒤 단계로. 이건 AI 성능 문제가 아니라 님의 계정·설정·결정이 필요하기 때문.
  • 생각 노트는 유도 질문 없이 빈 종이 방식 (placeholder 문구 하나만).
  • 펜 손글씨+텍스트 변환은 최종 목표로 확정하되 순서는 뒤로. 지금은 데이터 자리만 마련.
  • 화면은 폰과 태블릿 모두 쓸 수 있도록 반응형으로 설계.
  • 데이터 접근은 StorageService 한 곳으로 모은다(화면·상태는 이 클래스를 통해서만 데이터에 접근). 향후 로컬(Hive)→클라우드 마이그레이션을 쉽게 하기 위함.
  • Hive 모델에 필드/모델을 추가할 때는 로컬에서 dart run build_runner build --delete-conflicting-outputs를 돌려 .g.dart를 다시 생성한다. build_runner가 "0 outputs"를 내면 캐시 문제이므로 dart run build_runner clean 후 다시 빌드한다. (GitHub 웹에서 .g.dart를 손으로 고치는 건 오류가 잘 나므로 피한다)
  • 새 Hive 모델을 만들 때는 typeId가 겹치지 않게 한다. 현재 사용 중: Task=0, Note=1, NoteStatus=2, Reflection=3, Habit=4, MorningSession=5, Project=6. 다음은 7부터. (필드만 추가하는 경우는 새 HiveField 번호만 늘리면 되고 typeId는 그대로)
  • 새 typeId를 추가한 뒤, 이미 앱을 설치했던 기기에서는 Hive 어댑터 오류가 날 수 있으므로 flutter clean 후 재빌드하거나 앱을 재설치한다.
  • GitHub 웹에서 코드를 수정한 경우, 로컬에서 빌드하기 전에 git pull origin main으로 먼저 받아온다. pull이 fetch만 되고 병합이 안 되거나 로컬이 뒤처지면 git fetch origin 후 git reset --hard origin/main으로 강제 동기화한다(로컬 변경분은 사라지니 주의).
  • git 커밋 전 사용자 정보 설정이 필요하면 git config --global user.name "lemon6243" / git config --global user.email "93980574+lemon6243@users.noreply.github.com".
  • 시간 측정은 시작 시각을 저장해 DateTime.now()와의 차이로 계산한다(초를 직접 세는 Timer.periodic은 백그라운드에서 멈추므로 UI 갱신용으로만 사용).
  • 이미지 등 에셋은 assets/ 아래(예: assets/growth/)에 두고 pubspec.yaml의 assets:에 폴더를 등록한다. 설계 문서 등 코드가 아닌 문서는 docs/에 둔다(앱 빌드에 포함되지 않음).
